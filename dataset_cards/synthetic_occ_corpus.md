# Dataset Card: TAKTKRONE-I Synthetic OCC Corpus

## Dataset Summary

The TAKTKRONE-I Synthetic OCC Corpus is a purpose-built dataset for training and evaluating metro Operations Control Center language models. It contains synthetic operational scenarios, reasoning traces, and instruction tuning samples designed to support disruption diagnosis, headway regulation, dispatch decision support, and service recovery planning.

**Important:** This is NOT a generic transportation QA dataset. It is designed to reflect how a real metro OCC thinks, writes, escalates, and evaluates operational options.

## Dataset Structure

### Data Instances

Each sample contains:

```json
{
  "scenario_id": "scn_mta_bunching_001",
  "timestamp_utc": "2026-03-27T17:30:00Z",
  "operator": "mta_nyct",
  "scenario_type": "bunching",
  "incident_briefing": {...},
  "network_state": {...},
  "reasoning_trace": {...},
  "instruction_sample": {
    "messages": [
      {"role": "system", "content": "..."},
      {"role": "user", "content": "..."},
      {"role": "assistant", "content": "..."}
    ],
    "task_type": "HCS"
  },
  "taxonomy": {...},
  "metadata": {...},
  "ground_truth": {...}
}
```

### Data Fields

**Core Fields:**
- `scenario_id`: Unique identifier
- `timestamp_utc`: Scenario time (ISO 8601)
- `operator`: Transit operator code
- `scenario_type`: High-level scenario category
- `incident_briefing`: Human-readable incident summary
- `network_state`: Detailed network state data
- `reasoning_trace`: Full OCC reasoning process
- `instruction_sample`: Formatted for instruction tuning
- `taxonomy`: Multi-dimensional classification
- `metadata`: Generation and validation metadata
- `ground_truth`: Gold labels for evaluation

**Taxonomy Fields:**
- `incident_type`: SIG, THS, TDS, TCG, etc.
- `severity`: low, medium, high, critical
- `difficulty`: easy, medium, hard, expert
- `split`: train, validation, test
- `telemetry_completeness`: full to minimal
- `output_mode`: text_summary, action_recommendation, etc.

### Data Splits

| Split | Count | Purpose |
|-------|-------|---------|
| train | 60,800 | Model training |
| validation | 7,600 | Hyperparameter tuning |
| test | 7,600 | Final evaluation |

**Split Constraints:**
- No scenario templates shared between train and test
- Operator stratification maintained
- Difficulty balanced across splits
- No duplicate or near-duplicate samples across splits

## Dataset Creation

### Generation Methodology

1. **Rule-Based Template Generation**
   - Parameterized scenario templates
   - Controlled randomization of parameters
   - Topology-aware generation

2. **Simulation-Based Generation**
   - Delay propagation simulation
   - Multi-train interaction modeling
   - Terminal turnback simulation

3. **Quality Assurance**
   - Automated schema validation
   - Topology consistency checking
   - Plausibility verification
   - Human review sampling

### Source Data

- **Network Topology:** Derived from public GTFS data
- **Operational Patterns:** Based on published service guides
- **Incident Types:** Informed by public service advisories
- **Procedures:** Based on publicly available transit operations literature

No proprietary operator data was used. All scenarios are synthetic.

### Annotations

- **Automatic Labels:** Generated by templates and simulation
- **Quality Scores:** Automated quality scoring
- **Human Review:** 1% sample human reviewed

### Personal and Sensitive Information

This dataset contains NO:
- Personal identifiable information (PII)
- Real passenger data
- Actual incident records
- Proprietary operational data

All content is synthetic.

## Considerations for Using the Data

### Social Impact

This dataset is designed to support research on AI-assisted transit operations. Potential positive impacts:
- Improved transit service reliability
- Better disruption response
- Enhanced operator decision support

### Discussion of Biases

**Known Limitations:**
- Heavy-rail metro systems only (no light rail, bus, commuter rail)
- English language only
- US and UK operator focus
- Synthetic data may not capture all real-world complexity

**Mitigation:**
- Multiple operators included for diversity
- Various incident types and severities
- Edge cases and adversarial examples included

### Other Known Limitations

- Scenarios are synthetic, not historical incidents
- Some operational nuances may be missing
- Network topology simplified in some cases
- Crew-related constraints not fully modeled
